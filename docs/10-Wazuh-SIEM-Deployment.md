# 1. Overview

This document describes the deployment of the Wazuh Security Information and Event Management (SIEM) platform as part of the **SIEM Monitoring Home Lab** project.

The deployment was performed using the native installation method on Ubuntu Server rather than Docker in order to gain a deeper understanding of the platform architecture and the interaction between its core components.

The completed deployment includes:

- Wazuh Indexer
- Wazuh Manager
- Filebeat
- Wazuh Dashboard

Together, these components provide a centralized platform capable of collecting, indexing, storing, and visualizing security telemetry generated throughout the lab environment.

Unlike a basic installation guide, this document focuses on the engineering decisions, deployment process, infrastructure preparation, service validation, and troubleshooting activities required to build a fully operational SIEM platform.

---

# 2. Implementation Objectives

The primary objectives of this deployment were:

- Deploy the complete Wazuh SIEM platform using the native Linux packages.
- Configure secure TLS communication between all platform components.
- Implement centralized log collection and indexing.
- Validate secure communication between the Manager, Indexer, Filebeat, and Dashboard.
- Verify that security alerts generated by the Manager were successfully indexed and displayed through the Dashboard.
- Produce an enterprise-style deployment that serves as the foundation for future endpoint onboarding, detection engineering, threat hunting, and attack simulation exercises.

---

# 3. Target Architecture

The SIEM platform was deployed on a single Ubuntu Server hosting all core Wazuh services.

The deployment architecture consists of four primary components:

| Component | Purpose |
|-----------|---------|
| Wazuh Manager | Collects, analyzes, and processes security events from monitored endpoints. |
| Wazuh Indexer | Stores indexed security data and provides fast search capabilities using OpenSearch. |
| Filebeat | Collects alerts generated by the Manager and securely forwards them to the Indexer. |
| Wazuh Dashboard | Provides a web-based interface for monitoring, investigation, visualization, and administration. |

The overall data flow within the platform follows the sequence below:

```text
Endpoints
      │
      ▼
Wazuh Manager
      │
      ▼
Filebeat
      │
      ▼
Wazuh Indexer
      │
      ▼
Wazuh Dashboard
```

This architecture provides centralized security monitoring while maintaining secure TLS communication between every component.

# 5. Deployment Strategy Evaluation

Wazuh supports multiple deployment methods, including Docker containers, virtual appliances, and native package installation.

Before beginning the deployment, each option was evaluated against the objectives of this project.

The primary goal was not simply to deploy a working SIEM platform, but to understand how every Wazuh component communicates, integrates, and operates within a Linux environment.

---

## Docker Deployment

Docker provides a rapid deployment method by automatically configuring the Wazuh Manager, Indexer, Dashboard, and supporting services.

Advantages include:

- Rapid deployment
- Simplified upgrades
- Easy rollback
- Consistent configuration

However, containerization abstracts much of the platform's internal architecture, limiting exposure to service dependencies, certificate management, Filebeat configuration, and troubleshooting.

---

## Native Deployment

Native installation requires every component to be installed and configured individually.

This includes:

- Repository configuration
- Package installation
- TLS certificate deployment
- Filebeat configuration
- Service management
- Platform validation

Although this approach requires additional effort, it provides significantly greater understanding of the platform architecture and deployment process.

---

## Selected Deployment Strategy

Native deployment was selected because it aligns more closely with the educational objectives of this home lab.

Deploying each component manually provides practical experience with:

- Linux service management
- TLS certificate implementation
- Log forwarding architecture
- OpenSearch integration
- Wazuh component dependencies
- Enterprise troubleshooting methodologies

This approach transformed the deployment into an engineering exercise rather than a simple software installation.

---

# 12. Deployment Challenges and Lessons Learned

Although the deployment was ultimately successful, several technical challenges were encountered throughout the implementation.

Each issue was investigated individually, the root cause was identified, corrective actions were applied, and the platform was validated before continuing with the deployment.

Resolving these issues provided valuable practical experience with Linux system administration, service dependencies, and enterprise troubleshooting methodologies.

---

## 12.1 Infrastructure Preparation

Before beginning the deployment, an unrelated Docker container repeatedly restarted because of an invalid Docker Compose configuration.

Although the issue did not directly affect Wazuh, it was resolved before deployment to eliminate unnecessary background errors and establish a stable operating environment.

This reinforced the importance of validating existing infrastructure before introducing new enterprise services.

---

## 12.2 Certificate Management

During the deployment, the Wazuh Indexer initially failed to start because the generated TLS certificates did not have the required ownership and permissions. After reviewing the service logs, the certificate permissions were corrected, allowing the service to initialize successfully.

---

## 12.3 Filebeat Configuration

During Filebeat configuration, an incorrect index template was initially downloaded.

The issue was identified during validation and corrected by replacing the template with the official version before enabling the service.

This emphasized the importance of verifying configuration resources before deployment rather than assuming downloaded files are correct.

---

## 12.4 Incremental Validation

Every major component was validated immediately after deployment before introducing additional dependencies.

This incremental deployment strategy significantly simplified troubleshooting because each problem could be isolated to a single component rather than the complete platform.

The approach also reduced deployment risk by ensuring that problems were corrected before they propagated throughout the infrastructure.

---

## 12.5 Lessons Learned

The deployment reinforced several important engineering principles:

- Validate existing infrastructure before deployment.
- Deploy services incrementally rather than simultaneously.
- Verify every dependency before continuing.
- Correct certificate permissions are as important as certificate generation.
- Structured troubleshooting is more effective than trial-and-error debugging.
- End-to-end validation should always conclude infrastructure deployments.
